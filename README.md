# simvek - semantic similarity search for amino acid sequences

simvek embeds a given query sequence with a transformer model and finds the nearest neighbours in embedding space, returning semantically similar sequences without relying
on traditional alignment methods like BLAST. 

Vectors are indexed with [HNSW](https://github.com/jean-pierreBoth/hnswlib-rs) (Hierarchial Navigable Small World) for fast approximate search of neighbouring vectors. 

Embeddings and sequences are persisted in [sled](https://github.com/spacejam/sled) key-value stores for fast retrieval via a shared id.

## Architecture
### Crates
#### core-db
- HNSW index and sled key-value storage for sequences and embeddings

#### ml-models
- Embedder for amino acid sequences
- Note: Embeddings are generated by `scripts/embed_query.py` (one vector per sequence) The default model is `Rostlab/prot_bert` (1024-dim).

#### cli
- Command line interface, handles build and query subcommands


## Requirements
- Rust (2021 edition)
- Python $>=$ 3.13
- uv

For setup, run `uv sync` and make python3 is in `$PATH`. This is needed for the embedding script.
**Note:** the embedding script is called via a subprocess which assumes that your python binary can be run with the `python3` command.

## Usage

For the concrete flags and descriptions, consult the help page:

**Using cargo:**
```
cargo run -p cli -- --help
cargo run -p cli build --help
cargo run -p cli query --help
```

**Using the binary:**
```
cli --help
cli build --help
cli query --help
```

### Examples
#### Build from a FASTA file
```sh
cargo run -p cli -- build \
  --fasta sequences.fasta \
  --record-type protein \
  --model Rostlab/prot_bert \
  --dim 1024
```

Use `--fresh` for a rebuild, i.e. to wipe the database before building.

#### Query 5 nearest neighbours

```sh
cargo run -p cli -- query \
  --query "SAISFHSGYSGLVATVSGSQQTLVVALNSDLGNPGQVASGSFSEAVNASNGQVRVWR" \
  --record-type protein \
  --top-k 5 \
  --output json
```

**Output formats:** `plain` (default) or `json`.

### Advanced flags: HNSW tuning

#### Index quality (higher = better, slower build)
**Flag:** `--ef-construction`
**Default:** 200
#### Query accuracy vs speed (higher = better, slower search)
**Flag:** `--ef-search`
**Default:** 50

## How does it work
```
data/
    sequence_db/   # sled tree: internal_id -> FastaRecord
    vector_db/     # sled tree: internal_id -> embedding (Vec of f32)
```

**Note:** No half precision (f16) support yet

### Build
1. The sequences from the fasta are saved in `sequence_db` and individual internal ids are assigned
2. The sequences are embedded and saved in `vector_db` using the same individual internal ids

### Query
1. The HNSW index is rebuilt in-memory from `vector_db` on startup.
2. The k nearest-neighbours are approximated in the HNSW using L2 distance
3. The according internal ids are used to query the according sequences from `sequence_db`
